# Phases 17-19+ Implementation Plan

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development (recommended) or superpowers:executing-plans to implement this plan task-by-task. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** evolve the app from single-restaurant consultant-first delivery into a multi-client consulting SaaS with deterministic automation and production hardening.

**Architecture:** Phase 17 adds multi-client access without breaking the current n=1 model: `restaurants.owner_id` remains valid, and a new relationship table grants consultants access to additional restaurants. Phase 18 adds deterministic recommendations/templates before any LLM. Phase 19+ hardens security, observability and operations after the functional surface is stable.

**Tech Stack:** Next.js App Router, TypeScript, Supabase/Postgres/RLS, Vitest, Testing Library, Playwright, Pino logger.

## Phase Boundaries

### Phase 17: Multi-Client Consultant Workspace

Build portfolio management for a consultant who works with several restaurant clients.

Must not rewrite all product actions in one pass. The first pass adds:

- Relationship table `consultant_restaurants`.
- Server-side active client resolution for consultant workspace.
- Client portfolio list in `/consultant`.
- Selector links to work on a client through controlled server actions/cookies.
- Tests that prove no restaurant id is trusted from arbitrary client input.

Out of scope for the first pass:

- End-client self-service data entry.
- Billing per client.
- Full migration away from `restaurants.owner_id`.
- Rewriting every existing page to use consultant relationship checks.

### Phase 18: Deterministic Automation Before AI

Add automation that is explainable and testable:

- Narrative templates for common financial readings.
- Alert rules for report readiness and delivery follow-up.
- Optional LLM only after deterministic templates exist and only with stored audit trail.

Out of scope initially:

- Free-form AI recommendations shown directly to clients.
- Model-based anomaly detection where thresholds suffice.

### Phase 19+: Production Hardening

Close operational risk:

- Multi-tenant isolation tests.
- RLS audit docs and SQL checks.
- Backup/export per restaurant.
- Health checks expanded with dependency status.
- Rate limiting for write endpoints/API routes.
- Error boundaries and operational runbook.

## Task 1: Phase 17.1 Schema Foundation

**Files:**
- Create: `supabase/migrations/<timestamp>_consultant_restaurants.sql`
- Modify: `docs/ai/T02-base-de-datos.md`

- [ ] **Step 1: Write the migration**

```sql
create table if not exists public.consultant_restaurants (
  id uuid primary key default gen_random_uuid(),
  consultant_user_id uuid not null references auth.users(id) on delete cascade,
  restaurant_id uuid not null references public.restaurants(id) on delete cascade,
  role text not null default 'CONSULTANT' check (role in ('OWNER', 'CONSULTANT', 'VIEWER')),
  status text not null default 'ACTIVE' check (status in ('ACTIVE', 'PAUSED', 'REVOKED')),
  created_at timestamptz not null default now(),
  updated_at timestamptz not null default now(),
  unique (consultant_user_id, restaurant_id)
);

create index if not exists idx_consultant_restaurants_consultant
  on public.consultant_restaurants (consultant_user_id, status);

create index if not exists idx_consultant_restaurants_restaurant
  on public.consultant_restaurants (restaurant_id, status);

alter table public.consultant_restaurants enable row level security;

drop policy if exists "consultants can read own client links" on public.consultant_restaurants;
create policy "consultants can read own client links"
  on public.consultant_restaurants for select
  to authenticated
  using (consultant_user_id = auth.uid());

drop policy if exists "restaurant owners can read client links" on public.consultant_restaurants;
create policy "restaurant owners can read client links"
  on public.consultant_restaurants for select
  to authenticated
  using (
    restaurant_id in (
      select id from public.restaurants where owner_id = auth.uid()
    )
  );

grant select on public.consultant_restaurants to authenticated;
```

Tests must verify:

- It excludes `PAUSED` and `REVOKED`.

- [ ] **Step 2: Implement action**

```ts
export async function getConsultantPortfolio(): Promise<ActionResponse<ConsultantClientSummary[]>> {
  const supabase = await createClient()
  const { data: { user } } = await supabase.auth.getUser()
  if (!user) return { success: false, error: 'No autenticado.' }

  const [ownedRes, linkedRes] = await Promise.all([
    supabase.from('restaurants').select('id, name, consultant_name').eq('owner_id', user.id),
    supabase
      .from('consultant_restaurants')
      .select('restaurant_id, role, status, restaurants(id, name, consultant_name)')
      .eq('consultant_user_id', user.id)
      .eq('status', 'ACTIVE'),
  ])

  if (ownedRes.error || linkedRes.error) return { success: false, error: 'No se pudo cargar la cartera.' }

  // Merge by restaurant id. Owned rows win with role OWNER.
}
```
